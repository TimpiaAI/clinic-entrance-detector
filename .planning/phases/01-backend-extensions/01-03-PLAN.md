  <name>Task 1: Create api/sleep_guard.py with wake-lock activation/deactivation endpoints</name>
  <files>api/sleep_guard.py, tests/test_sleep_guard.py</files>
  <behavior>
    - Test: POST /wake-lock when not active activates wake lock and returns {"status": "active"}
    - Test: POST /wake-lock when already active returns {"status": "already_active"}
    - Test: POST /wake-lock/release when active deactivates and returns {"status": "inactive"}
    - Test: POST /wake-lock/release when not active returns {"status": "already_inactive"}
    - Test: wake_lock_status() returns {"active": False} when no lock held
    - Test: wake_lock_status() returns {"active": True} when lock is held
    - Test: If keep.presenting() Mode.active is False, returns {"status": "failed"} and cleans up
  </behavior>
  <action>
    1. Install wakepy: `pip install wakepy==1.0.0`

    2. Create `api/sleep_guard.py` with:
       - `from fastapi import APIRouter`
       - `from wakepy import keep`
       - Module-level globals: `_keep_ctx = None`, `_mode = None`
       - `router = APIRouter(prefix="/api/system", tags=["system"])`
       - `POST /wake-lock` endpoint:
         - If `_mode is not None and _mode.active`: return `{"status": "already_active"}`
         - `_keep_ctx = keep.presenting()`
         - `_mode = _keep_ctx.__enter__()`
         - If not `_mode.active`: exit context, clean up globals, return `{"status": "failed", "detail": "Could not activate sleep prevention"}`
         - Return `{"status": "active"}`
       - `POST /wake-lock/release` endpoint:
         - If `_keep_ctx is None`: return `{"status": "already_inactive"}`
         - `_keep_ctx.__exit__(None, None, None)`
         - Set both globals to None
         - Return `{"status": "inactive"}`
       - `wake_lock_status() -> dict` function (not an endpoint, called by DashboardState):
         - Return `{"active": _mode is not None and _mode.active}`

    3. Create `tests/test_sleep_guard.py`:
       - Mock `wakepy.keep.presenting` to return a mock context manager
       - Mock Mode object with `.active = True` (success) and `.active = False` (failure)
       - Test all behaviors listed above
       - Use `importlib.reload` or reset globals between tests

    Pattern reference: See RESEARCH.md "Pattern 5: Sleep Guard with Programmatic Context Manager Control" for exact implementation.
  </action>

  <name>Task 2: Add video serving endpoint with HTTP 206 range support and StaticFiles mount</name>
  <files>dashboard/web.py, .env, config.py, tests/test_video_serve.py</files>
  <behavior>
    - Test: GET /api/videos/video1.mp4 without Range header returns 200 with full file
    - Test: GET /api/videos/video1.mp4 with Range: bytes=0-1023 returns 206 with Content-Range header
    - Test: GET /api/videos/nonexistent.mp4 returns 404
    - Test: GET /api/videos/malicious.mp4 returns 404 (not in ALLOWED_VIDEOS whitelist)
    - Test: Range beyond file size returns 416
  </behavior>
  <action>
    1. Add `VIDEO_DIR` to config.py Settings and .env:
       - In `config.py` Settings dataclass, add: `VIDEO_DIR: str = ""`
       - In `load_settings()`, add: `VIDEO_DIR=_env_str("VIDEO_DIR", ""),`
       - In backward-compatible constants section: `VIDEO_DIR = SETTINGS.VIDEO_DIR`
       - In `.env`, append: `VIDEO_DIR=` (empty means project root)

    2. In `dashboard/web.py`, inside `create_dashboard_app()`, add video serving endpoint:
       - Import `Path` from pathlib (already imported), `os`
       - Determine VIDEO_DIR: `video_dir = Path(os.getenv("VIDEO_DIR", "")) or Path(__file__).resolve().parent.parent`
         - If VIDEO_DIR env is empty string, fall back to project root
       - Define `ALLOWED_VIDEOS = {f"video{i}.mp4" for i in range(1, 9)}`
       - Add `GET /api/videos/{filename}` endpoint:
         - If filename not in ALLOWED_VIDEOS: raise HTTPException(404)
         - If file not on disk: raise HTTPException(404, "Video file missing")
         - Get file_size via stat
         - Parse Range header: if None, return full file as Response(content=bytes, media_type="video/mp4", headers={"Accept-Ranges": "bytes", "Content-Length": str(size)})
         - If Range present: parse "bytes=START-END", clamp end to min(end, file_size-1), validate range, read byte range, return Response(content=data, status_code=206, media_type="video/mp4", headers={"Content-Range": f"bytes {start}-{end}/{file_size}", "Accept-Ranges": "bytes", "Content-Length": str(length)})
         - If range invalid: raise HTTPException(416, "Range not satisfiable")

    3. Add StaticFiles mount for future Vite frontend (BACK-01):
       - At the END of `create_dashboard_app()`, just before `return app`:
       - `frontend_dist = Path(__file__).resolve().parent.parent / "frontend_dist"`
       - `if frontend_dist.is_dir(): app.mount("/", StaticFiles(directory=str(frontend_dist), html=True), name="frontend")`
       - This is a conditional mount -- only activates when `frontend_dist/` exists (after Phase 2 builds)
       - Import `StaticFiles` at top: `from fastapi.staticfiles import StaticFiles`

    4. Create `tests/test_video_serve.py`:
       - Use FastAPI TestClient
       - Create a temporary video file (small bytes, just for testing range headers)
       - Mock the VIDEO_DIR / ALLOWED_VIDEOS to point to the temp file
       - Test all behaviors listed above
       - Verify 206 response has correct Content-Range header format

    Pattern reference: See RESEARCH.md "Pattern 4: Video Serving with HTTP 206 Range Requests" for exact implementation.

    CRITICAL: Do NOT use StaticFiles for video serving (Pitfall 3). Use custom endpoint with Range header parsing.
    CRITICAL: Mount StaticFiles AFTER all API routes (FastAPI route matching is first-match).
    CRITICAL: ALLOWED_VIDEOS whitelist prevents path traversal attacks.
